## 1. プロジェクト概要

### 1.1 目的と背景

AppGeniusは、ClaudeCodeを活用しながら非技術者でも自分の脳内にあるアイデアを形にすることができるアプリケーションを、自然言語だけで作成できる開発支援ソフトウェアです。日本のデジタル生産性を世界一に引き上げることを使命として進めているプロジェクトです。
主なターゲットユーザーは企業の経営者や意思決定者です。このAI時代において、経営者は自分の頭脳を代行したりシステム化することで効率化できることを理解していますが、従来は要件定義を行うのはプログラマーであったため、アイデアが適切に伝わらないという課題がありました。
AppGenius（ブルーランプ）を使うことで、AIが即座に要件定義を形にしながらモックアップを作成し、視覚的なフィードバックを提供しながらアイデアを形にしていくことができるため、これまでとは全く異なる開発体験を実現します。
 **設計中心のコード自動生成（Design-Centered Code Automation）**
   - 人間は設計思考に集中し、AIが実装を担当するパラダイム
   - 設計意図を明確に表現できるドキュメント形式を優先する

### 1.2 ファイルの状況について

AppGenius/refference/HinagoProject★3 ディレクトリには、それぞれのエージェントの処理が終わった成果物のファイルが入っています。これらのファイルは参考資料として活用できます。

### 1.3 型定義とAPIパスの管理戦略

プロジェクトでは型定義とAPIパスの一貫性を確保するために、以下のシンプルなルールを採用します：

#### 1.3.1 型定義ファイルの配置

- **フロントエンド**: `frontend/src/types/index.ts`
- **バックエンド**: `backend/src/types/index.ts`

#### 1.3.2 同期ルール

**基本原則：一方の/types/index.tsを更新したら、もう一方の/types/index.tsも必ず同じ内容に更新する**

この原則により、フロントエンドとバックエンド間で型定義とAPIパスの一貫性を常に保ちます。具体的には：

1. **変更の責任**
   - 型定義を変更した開発者は、両方のファイルを即座に同期させる責任を持つ
   - 1つのtypes/index.tsの更新は禁止。必ず1つを更新したらもう一つも更新その場で行う

2. **絶対に守るべき原則**
   - フロントエンドとバックエンドで異なる型を作らない
   - 同じデータ構造に対して複数の型を作らない
   - 新しいプロパティは必ずオプショナルとして追加
   - APIパスは必ずこのファイルで一元管理する
   - コード内でAPIパスをハードコードしない
   - 2つの同期されたtypes/index.tsを単一の情報源とする

3. **チーム間のコミュニケーション**
   - 型定義の重要な変更は事前にチーム間で共有する
   - APIに影響する変更は特に慎重に行い、両チームの合意を得る

このシンプルなルールにより、複雑な同期メカニズムや判断を必要とせず、常に一貫した型定義を維持します。

#### 1.3.3 注意事項

以下の点に特に注意して型定義の一貫性を確保してください：

1. **同期ミスの防止**
   - コミット前に必ずフロントエンドとバックエンドの型定義ファイルの内容が一致していることを確認
   - 型定義の不一致はランタイムエラーの原因となるため特に注意が必要
   - CI/CDパイプラインに型定義の一致を確認するステップを追加することを推奨

2. **型定義変更のレビュープロセス**
   - 型定義変更を含むPRでは、両方のファイルが更新されていることをレビュー時に確認
   - API関連の型変更は、インターフェースの破壊的変更を避けるため、後方互換性を確保

3. **リファクタリング時の考慮事項**
   - 大規模なリファクタリングを行う際は、型定義ファイルの変更を最初に行い、依存関係の問題を早期に検出
   - 変更の影響範囲が大きい場合は、型定義の更新履歴を文書化しておくことで追跡を容易にする

  ### コード品質の原則

  - 問題の根本原因を特定し、正面から解決する
  - ハードコードやモックデータによる一時しのぎの解決策を提案しない

  - 「魔法のような解決策」や「ブラックボックス」を避ける
